# Repo Structure

Noted is **one Next.js app** plus **one Convex backend**, with a few flat top-level folders. There's no monorepo, no workspaces, no `packages/`. If you've been writing in a Turborepo, leave the cross-package import rules at the door.

## Top-level layout

```
noted-main/
├── app/                  # Next.js 16 App Router (routes, layouts, API endpoints)
├── components/           # All React components (UI primitives + features)
├── convex/               # Convex backend — schema, queries, mutations, actions
├── hooks/                # Custom React hooks AND Zustand stores
├── lib/                  # Shared utilities, AI agent code, analytics
├── public/               # Static assets served at the root URL
├── scripts/              # Repo scripts (sync-ai.mjs, etc.)
├── team-os/              # Team OS — PRDs, dossiers, research, ship-logs
├── .ai/                  # Skills + slash commands — source of truth for agents
├── .specify/             # Speckit framework (constitution, templates, scripts)
├── .claude/              # AUTO-GENERATED symlinks of .ai/ for Claude Code
├── .cursor/              # AUTO-GENERATED symlinks of .ai/ for Cursor
├── DESIGN.md             # Visual identity contract (Constitution §XVI)
├── CLAUDE.md             # Auto-generated symlink to .ai/INSTRUCTIONS.md
├── AGENTS.md             # Auto-generated symlink to .ai/INSTRUCTIONS.md
├── README.md             # Local dev setup
├── DEPLOYMENT.md         # How the app ships
├── TECH_STACK.md         # What each dependency is for
├── package.json          # npm (NOT pnpm)
├── tsconfig.json         # @/* alias mapped to repo root
├── tailwind.config.ts
└── next.config.ts
```

`.claude/` and `.cursor/` are gitignored and regenerated by `npm run sync-ai` (postinstall hook). Edit `.ai/`, never `.claude/` or `.cursor/`.

## `app/` — Next.js App Router

```
app/
├── layout.tsx                       # root layout — providers
├── globals.css                      # Tailwind + HSL custom-property tokens
├── (landing)/                       # marketing — public, no auth
│   ├── layout.tsx
│   ├── page.tsx                     # / — landing page
│   ├── _components/                 # landing-only components (private)
│   └── features/<slug>/page.tsx
├── (main)/                          # authenticated app
│   ├── layout.tsx                   # auth check + sidebar shell
│   ├── _components/                 # main-app-only components (private)
│   └── (routes)/
│       ├── documents/page.tsx       # /documents
│       ├── documents/[documentId]/page.tsx
│       ├── coworkers/page.tsx
│       └── files/page.tsx
├── (public)/                        # publicly-shared documents (no auth)
│   └── (routes)/preview/[documentId]/page.tsx
└── api/
    ├── edgestore/[...edgestore]/route.ts
    └── ai/
        ├── chat/route.ts
        └── coworker/route.ts
```

Three route groups: **`(landing)`** (marketing), **`(main)`** (authenticated app), **`(public)`** (anonymous shareable views). Underscore-prefixed folders (`_components/`) are private — Next.js excludes them from routing and only the parent route group can import from them.

See the `nextjs-app-router` skill for what goes inside.

## `components/` — all UI

```
components/
├── ui/                              # shadcn primitives — DO NOT MODIFY
│   ├── button.tsx
│   ├── dialog.tsx
│   ├── popover.tsx
│   └── ...                          # 17 primitives shipped today
├── providers/                       # client-tree providers
│   ├── convex-provider.tsx
│   ├── theme-provider.tsx
│   ├── modal-provider.tsx
│   └── amplitude-provider.tsx
├── modals/                          # global modals
│   ├── settings-modal.tsx
│   ├── cover-image-modal.tsx
│   └── ...
├── coworker/                        # feature folder — coworker chat
│   ├── coworker-chat.tsx
│   ├── coworker-message.tsx
│   ├── coworker-floating-chat.tsx
│   └── coworker-context-selector.tsx
└── editor.tsx                       # top-level features that don't merit a folder
└── search-command.tsx
└── toolbar.tsx
└── ...
```

The split:

- **`components/ui/`** — vendored shadcn primitives. Add new ones via `npx shadcn add`; never modify existing files (drift makes upgrades painful).
- **`components/providers/`** — client tree providers, mounted from `app/layout.tsx`.
- **`components/modals/`** — globally-mounted modals controlled by Zustand stores in `hooks/`.
- **`components/<feature>/`** — feature folders for clusters of related components. Use when 3+ components share a feature.
- **`components/<single>.tsx`** — one-off components used in one place don't need a folder.

See the `design-system` and `component-composition` skills.

## `convex/` — backend

```
convex/
├── _generated/                      # AUTO-GENERATED — do not edit
├── schema.ts                        # defineSchema, all tables, all indexes
├── documents.ts                     # query + mutation handlers per domain
├── aiSettings.ts                    # public + internal handlers (no Node)
├── aiSettingsActions.ts             # 'use node' file — actions only
├── coworkerMessages.ts
├── files.ts
├── squadAgents.ts
└── storage.ts
```

camelCase file names (matches Convex's generated `api.<file>.<fn>` surface). One file per domain noun, plural. Action-only modules end in `Actions.ts` and start with `"use node";`. See `convex-handlers` and `convex-schema`.

## `hooks/` — custom hooks AND Zustand stores

```
hooks/
├── use-search.tsx                   # Zustand: search modal open state
├── use-cover-image.tsx              # Zustand: cover image modal state
├── use-settings.tsx                 # Zustand: settings modal state
├── use-coworker.ts                  # Zustand: coworker UI session state
├── use-coworker-config.ts           # Zustand: coworker layout config
├── use-file-picker.tsx              # Zustand: file picker modal state
├── use-tracked-upload.tsx           # Zustand: upload progress tracking
├── use-ai-settings.tsx              # custom hook (not a store)
├── use-origin.tsx                   # browser origin (window.location)
├── use-scroll-top.tsx               # scroll position helper
└── *.test.ts(x)                     # colocated tests
```

Two species in one folder, both prefixed `use-`:

- **Zustand stores** — single-file pattern, `create<T>(...)` exporting one hook (see `state-management` skill).
- **Custom React hooks** — wrap browser APIs (`use-origin`, `use-scroll-top`) or compose Convex hooks (`use-ai-settings`).

The folder is flat. No subdirectories.

## `lib/` — utilities and the AI agent

```
lib/
├── utils.ts                         # the `cn()` helper
├── analytics.ts                     # Amplitude wrapper + track* helpers
├── ai-models.ts                     # AI provider/model constants and lookups
├── edgestore.ts                     # EdgeStore client config
├── server-side-transport.ts         # SSR-safe transport for AI SDK
└── agent/                           # AI agent code (NOT routes — those are in app/api/)
    ├── prompts/
    │   └── squad-prompts.ts
    └── tools/
        └── workspace.ts
```

Anything that's:

- Pure utility shared across the app → `lib/<thing>.ts`
- AI-agent prompts and tool definitions → `lib/agent/prompts/*` and `lib/agent/tools/*`
- Single-component helper → `<feature>.utils.ts` next to the component (NOT here — see `file-naming`)

See the `agent-tool-authoring` skill for the inside of `lib/agent/`.

```
team-os/
├── ROSTER.md

├── CONTRIBUTING.md
├── product/                         # PRDs, RFCs
├── engineering/                     # constitution, runbooks, tech notes

├── support/                         # FAQs, talking points, escalation paths
├── growth/                          # marketing, growth experiments
├── team/                            # rituals, playbooks (claude-playbook.md)

```

Cross-functional area, **first-class part of the repo**. Anything that helps the team understand what we're building, why, and how it's used goes here. The /ship-log command appends to a feature's `ship-log.md` whenever a PR merges.

## `.ai/` — skills + commands (source of truth)

```
.ai/
├── INSTRUCTIONS.md                  # the agent system prompt — symlinked from CLAUDE.md / AGENTS.md
├── CONTRIBUTING.md                  # how to add new skills/commands
├── CREDITS.md
├── skills/<name>/SKILL.md           # one folder per skill
└── commands/<name>.md               # one file per slash command
```

`.claude/` and `.cursor/` are auto-generated from `.ai/` by `scripts/sync-ai.mjs`. Always edit `.ai/`. After adding or renaming a skill/command, run `npm run sync-ai` (or `npm install` — the postinstall hook handles it).

## `.specify/` — Speckit

```
.specify/
├── memory/constitution.md           # condensed mirror of team-os/engineering/constitution.md
├── scripts/bash/<name>.sh
└── templates/                       # spec, plan, tasks templates (for /speckit.* commands)
```

The constitution at `.specify/memory/constitution.md` is the version Speckit's commands read; the long-form version lives at `team-os/engineering/constitution.md`. Keep them in sync — when one changes, update the other in the same PR.

## `scripts/` — repo scripts

```
scripts/
├── sync-ai.mjs                      # materializes .ai/ → .claude/ + .cursor/ symlinks
└── ...
```

For one-off "do this once" scripts (data migrations, batch updates), use this folder. For Convex migrations, the migration is a Convex action invoked manually.

## Decision tree — where does this file go?

```
Is it a database table or a server-side handler?
  └─ convex/<domain>.ts (camelCase)
     └─ if it needs Node APIs: convex/<domain>Actions.ts with "use node"

Is it a Next.js route, layout, or API endpoint?
  └─ app/...
     └─ marketing surface? app/(landing)/...
     └─ authenticated app surface? app/(main)/...
     └─ publicly shared (no auth)? app/(public)/...
     └─ HTTP-specific (webhook, streaming)? app/api/<feature>/route.ts

Is it a React component?
  └─ shadcn primitive? components/ui/<name>.tsx (rare — most come from npx shadcn)
  └─ provider for the client tree? components/providers/<name>.tsx
  └─ globally-mounted modal? components/modals/<name>.tsx
  └─ part of a feature cluster (3+ files)? components/<feature>/<file>.tsx
  └─ one-off used in one route group? <route-group>/_components/<file>.tsx
  └─ one-off used app-wide? components/<file>.tsx

Is it a custom hook or a Zustand store?
  └─ hooks/use-<thing>.tsx

Is it a utility used across multiple features?
  └─ lib/<thing>.ts

Is it a small helper used by exactly one component?
  └─ <feature>.utils.ts NEXT to the component, not in lib/

Is it documentation about the product, team, or process?
  └─ team-os/<area>/<file>.md

Is it a skill or slash command?
  └─ .ai/skills/<name>/SKILL.md  or  .ai/commands/<name>.md
     (then run `npm run sync-ai`)
```

## Path alias

Single alias: `@/*` → repo root, defined in `tsconfig.json`. Use it for any cross-folder import:

```typescript
// ✅
import { Button } from "@/components/ui/button";
import { useSearch } from "@/hooks/use-search";
import { api } from "@/convex/_generated/api";
import { cn } from "@/lib/utils";

// ❌ relative path traversal
import { Button } from "../../../components/ui/button";
```

Inside the same folder, prefer relative (`./sibling`) — keeps refactors local.

## What does NOT exist (and why)

| Concept | Status |
|---|---|
| `apps/` or `packages/` | No monorepo. One Next.js app + one Convex backend. |
| `services/` directory | Convex handlers ARE the data + business layer. No 3-layer split. |
| `dal/` directory | Same — Convex queries/mutations are the data layer. |
| `types/` directory | Convex generates `Doc<>` and `Id<>` for table types; component prop types live in the component file. No top-level types folder. |
| `index.ts` barrel files | Each consumer imports the specific file. Barrels add re-exports without value. |
| `__tests__/` directories | Tests are colocated as `<source>.test.ts(x)` next to the source. |
| `pnpm-workspace.yaml` / `turbo.json` | npm only. No turbo. |

If you're tempted to add one of these, **don't** — match noted's smaller-and-flatter shape.

## Anti-patterns

```
❌ creating a top-level services/ or dal/ folder
   → put queries/mutations in convex/<domain>.ts

❌ creating a types/ directory
   → component prop types live in the component file
   → DB row types come from Doc<"table"> in @/convex/_generated/dataModel

❌ creating an index.ts barrel under each component folder
   → consumers import the specific file:
     import { CoworkerMessage } from "@/components/coworker/coworker-message";

❌ putting a one-off helper in lib/
   → if it's used by exactly one component, it goes in <feature>.utils.ts
     next to that component

❌ committing .claude/ or .cursor/ files manually
   → they're auto-generated by npm run sync-ai; edit .ai/ instead

❌ adding cross-app import rules
   → there are no apps; one Next.js app + one Convex backend
```

## Checklist for a new file

- [ ] Folder picked using the decision tree (not pattern-matched from heatseeker / a previous repo)
- [ ] File name follows the convention for its folder (kebab-case everywhere except `convex/`); see `file-naming`
- [ ] No new top-level folder added unless absolutely necessary (and never `services/` / `dal/` / `types/`)
- [ ] Cross-folder imports use `@/*`; same-folder imports use `./`
- [ ] If you added/renamed a skill or command, `npm run sync-ai` was run
